# Boise Public Works Intelligence System (PWIS)

**A flagship public-sector data analytics portfolio project.**
End-to-end infrastructure investment prioritization platform for a mid-size city government — from synthetic data generation through a scored prioritization model, interactive GIS maps, executive Streamlit dashboard, and scenario simulation engine.

> *"Infrastructure investment decisions made without data cost cities 3–5× more in the long run and undermine public trust when they appear arbitrary. PWIS makes those decisions systematic, auditable, and defensible."*

---

## What This System Does

PWIS answers the core question facing every Public Works director:

> **"Given a limited maintenance budget and hundreds of road segments in varying states of deterioration, which ones do we fix first — and how do we prove it?"**

It does this through five integrated components:

| Component | What It Does |
|-----------|-------------|
| **Prioritization Model** | Scores every road segment 0–100 using a weighted, explainable formula across condition, traffic, complaints, cost efficiency, and equity |
| **GIS Layer** | Four interactive Folium maps (condition, priority, complaints, executive multi-layer) |
| **Executive Dashboard** | Streamlit app with KPI cards, district comparison, budget scenarios, and scenario simulation |
| **Scenario Engine** | "What if" analysis: budget allocation, weight sensitivity, 5-year deferral costs, coverage analysis |
| **BI Maturity Layer** | Star schema DDL, DAX measures, Row-Level Security, Dev/Test/Prod governance, CI/CD design |

## Quick Start

### 1. Install dependencies
```bash
pip install -r requirements.txt
```

### 2. Generate synthetic data
```bash
python data/generate_data.py
```

### 3. Run the prioritization model
```bash
python models/prioritization.py
```

Sample output:
```
Priority Tier Distribution:
  High      :   13 segments (4.3%)
  Medium    :  139 segments (46.3%)
  Low       :  148 segments (49.3%)

Top Priority — SEG-0237, Emerald Dr, Downtown (Highway, CI=38, Score=66.97)
Recommended: Full Rehabilitation — Current CIP Cycle
```

### 4. Generate GIS maps
```bash
python gis/map.py
# Opens 4 HTML maps in docs/
```

### 5. Run scenario engine
```bash
python models/scenario_engine.py
```

### 6. Launch the dashboard
```bash
streamlit run app/streamlit_app.py
```
Opens at `http://localhost:8501`

## The Prioritization Model

### Formula
```
Priority Score (0–100) =
    Condition Severity × 0.35
  + Traffic Impact      × 0.25
  + Complaint Pressure  × 0.20
  + Cost Efficiency     × 0.12
  + Equity Modifier     × 0.08
```

### Why This Design

| Decision | Rationale |
|----------|-----------|
| **Weighted sum over ML** | Explainable to Council — every score decomposable. No labeled failure data available yet. |
| **Condition at 35%** | Most objective, defensible input. Required for FHWA reporting and federal grant applications. |
| **Complaint density (not count)** | Normalizes by lane-mile to prevent population-dense districts from dominating. Equity design choice. |
| **Equity modifier at 8%** | Small correction for systematic underinvestment in lower-income areas. Documented and disclosed to all users. |
| **Exponential condition curve** | Deterioration below CI=40 accelerates non-linearly; emergency repair costs 3–5× preventive. Linear model would underweight the worst segments. |

Full design rationale and documented tradeoffs: [`docs/model_logic.md`](docs/model_logic.md)

## BI Maturity Layer

This project includes full BI maturity documentation in [`docs/bi_maturity.md`](docs/bi_maturity.md):

### Star Schema (SQL DDL)
- `DIM_DATE`, `DIM_SEGMENT`, `DIM_DISTRICT`, `DIM_WORK_TYPE`, `DIM_WEATHER`
- `FACT_WORK_ACTIVITY`, `FACT_CONDITION`, `FACT_COMPLAINT`, `FACT_BUDGET`
- Grain definitions, indexes, SCD Type 2 support

### DAX Measures (30+)
- Infrastructure Condition Index (ICI)
- District Health Index (composite)
- YoY condition change
- Preventive vs. reactive maintenance ratio
- Complaint resolution SLA %
- Budget variance and coverage
- Maintenance backlog ($)

### Row-Level Security
- 5 security roles: `pwis_exec`, `pwis_district_N`, `pwis_ops`, `pwis_finance`, `pwis_readonly`
- Dynamic RLS using `USERPRINCIPALNAME()` + `PWIS_UserSecurity` lookup table
- Full testing protocol documented

### Governance: Dev / Test / Prod
| Environment | Data | RLS | Refresh | Approval |
|-------------|------|-----|---------|----------|
| DEV | Synthetic | Off | Manual | None |
| TEST | Synthetic + sample real | Full | Daily | Analytics Lead |
| PROD | Real (masked PII) | Full | Every 4h | Director + IT |

### CI/CD Pipeline (Conceptual)
```
Git PR → Automated checks (pytest, sqlfluff, DAX lint)
      → Analytics Lead review
      → pbi-tools deploy to DEV workspace
      → UAT sign-off
      → Promote to PROD via GitHub Actions
      → Azure Monitor post-deploy watch
```

## Key Design Principles

1. **Explainability over precision** — Every score is auditable. Council members can see exactly why Segment A outranked Segment B.
2. **Equity by design** — Complaint normalization and the equity modifier prevent vocal/wealthy communities from monopolizing maintenance resources.
3. **Graceful degradation** — The system runs with partial data. Missing fields produce confidence warnings, not failures.
4. **Human decision authority** — The model recommends; humans decide. Every recommendation is advisory.
5. **Production-ready architecture** — Synthetic data, real schema. Drop in real data and deploy.
